# System Guide

────────────────────────────────────────────────────────
[목적]
────────────────────────────────────────────────────────

이 문서는 AI 기반 개발 시스템의 전체 동작 방식을 정의한다.

이 시스템은 다음 구조를 따른다:

- Front 중심 개발
- 문서 기반 설계
- Phase 분리 실행
- Task 기반 작업 통제

---

────────────────────────────────────────────────────────
[Phase 정의]
────────────────────────────────────────────────────────

P1 — FRONT_MOCK  
UI/UX 설계 및 프론트 목업 구현 단계

P2 — STRUCTURE  
데이터 구조 및 DB 설계 단계

P3 — INTEGRATION  
API 및 실제 데이터 연결 단계

---

────────────────────────────────────────────────────────
[브랜치 전략]
────────────────────────────────────────────────────────

브랜치 구조:

main   ← 버전 확정 시점 (git tag: v1, v2, ...)
p1     ← P1 작업 브랜치 (FRONT_MOCK)
p2     ← P2 작업 브랜치 (STRUCTURE)
p3     ← P3 작업 브랜치 (INTEGRATION)

원칙:

- p1, p2, p3 는 항상 존재하는 영속 브랜치다
- main 은 버전 확정 시에만 merge 한다
- 버전 경계는 git tag 로 표시한다
- Phase 전환은 반드시 Transition Task 통과 후 실행한다

태그 규칙:

v{N}-p1-done      ← T7 통과 시점
v{N}-p2-done      ← T8 통과 시점
v{N}-p3-snapshot  ← T9 발생 시점 (개선 사이클 진입)
v{N}-p3-done      ← 버전 확정 직전
v{N}              ← main merge 완료 시점

[프론트 프롬프트 구조]
────────────────────────────────────────────────────────

프론트 목업은 단일 문서로 생성하지 않는다.

다음 문서 세트를 조합하여 생성한다:

- PROJECT_META
- PRD_UI_SUMMARY
- GLOBAL_RULES
- ROUTER_MAP
- STORYBOARD
- ROUTES/*
- JSON_CONVENTION
- GLOBAL_QA

────────────────────────────────────────────────────────
[상태 출력 규칙 — 강제]
────────────────────────────────────────────────────────

모든 AI 응답은 반드시 아래 블록으로 시작해야 한다.

[CURRENT STATE]

Phase: P? — NAME  
Task: T? — NAME  

Context:
- 현재 작업 대상
- 작업 목적

Next Action:
- 다음 수행 작업

이 블록 없이 작업을 수행하는 것은 금지한다.

────────────────────────────────────────────────────────
[실행 순서 규칙]
────────────────────────────────────────────────────────

모든 작업은 아래 순서를 따른다:

1. system.md 읽기
2. task-definition.md 읽기
3. Task 확인
4. 해당 task 문서 읽기
5. 입력 문서 로드
6. 출력 문서 생성/수정

────────────────────────────────────────────────────────
[핵심 규칙]
────────────────────────────────────────────────────────

1. Phase를 넘나들며 작업하지 않는다  
2. Task 정의를 벗어나지 않는다  
3. current 폴더만 수정한다  
4. specs / tasks는 수정하지 않는다  
5. 항상 상태 블록을 먼저 출력한다
